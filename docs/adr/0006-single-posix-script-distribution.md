# ADR-0006: One POSIX sh script, curl-installed from GitHub Releases, skill bundled

**Status**: Accepted

The executable is a single `#!/bin/sh` script (no bashisms, no build step; runtime dependencies are `sh`, `tee`, `ln`, `mkdir`, `curl`, `git`). `install.sh` fetched via curl places it at `~/.local/bin/teetap` without sudo, pinned to the latest GitHub Release by default and overridable via a version environment variable — releases, not `main`, are the audit point for what users execute. The agent skill ships in the same repository under `skills/teetap/` (SKILL.md required; EXAMPLES.md and future reference files are optional detail — SKILL.md must stand alone). **Recommended onboarding is the skills.sh ecosystem** (`npx skills add https://github.com/<repo> --skill teetap`), deliberately separating binary installation from skill onboarding. `teetap skill install` remains as the no-Node fallback and is **offline by default**: it copies the skill directory bundled next to the script (checkout), or writes the SKILL.md embedded in the binary itself — version-locked by construction; `--allow-fetch` downloads the full skill structure from the binary's own release tag as a single tarball (no per-file fetching, no manifest). A test keeps the embedded copy byte-identical to `skills/teetap/SKILL.md`, so the two sources cannot drift. Trade-off accepted: `npx skills add` tracks the default branch (freshest docs, possible skew against an older binary) while the fallback is exactly version-locked; the skill is documentation-only, so skew is harmless by design. The installer also ships a man page (`man teetap`) alongside the binary.

## Considered options

npm/pip packaging was rejected (adds a runtime ecosystem the tool explicitly does not depend on). Homebrew was deferred, not rejected — the curl installer is the v1 channel. Curling `main` directly was rejected because an unreviewed push would change what users run.
